# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A monorepo with two halves that share one data file:

1. **The registry** (`skills/`, `commands/`, `agents/`, `registry.json`,
   `scripts/build-registry.mjs`) тАФ the content. It holds three **resource types**:
   - **skill** тАФ a folder of prompt files + a `skill.json` manifest тЖТ installs to
     `.claude/skills/<name>/`.
   - **command** тАФ a single `.md` (Claude Code slash command, YAML frontmatter) тЖТ
     installs to `.claude/commands/<name>.md`.
   - **agent** тАФ a single `.md` (Claude Code subagent, YAML frontmatter) тЖТ installs
     to `.claude/agents/<name>.md`.
2. **The `skillpool` CLI** (`packages/skillpool/`) тАФ published to npm. A package
   manager that installs resources into a consuming project's `.claude/`.
3. **The `agenticpool` CLI** (`packages/agenticpool/`) тАФ a second, separately
   published npm package doing the same job as skillpool but with an Ink
   (React) full-screen TUI (tabs per resource type, pagination, multi-select)
   instead of skillpool's raw-readline picker. It forks skillpool's core logic
   (`config.ts`/`registry.ts`/`installer.ts`) rather than depending on it, but
   reads/writes the exact same `.claude/skills/manifest.json`, so installs and
   removals are interoperable between the two tools. Test locally the same way:
   `node packages/agenticpool/dist/index.js --registry . list`.

The two are decoupled by **`registry.json`**: an auto-generated flat index with a
`skills[]`, `commands[]`, and `agents[]` array, each listing every resource of that
type and the exact files it contains. The CLI fetches this index from a GitHub raw URL
at runtime and downloads each listed file by raw URL тАФ it never uses the GitHub API and
never hardcodes the resource list. **Consequence: adding resources never requires a CLI
release.** Add a folder/file, regenerate the index, push.

**Name collisions** across types are allowed (e.g. skill `code-reviewer` + agent
`code-reviewer`). The CLI and the install manifest namespace every resource by type as
`type:name` (`skill:code-reviewer`). A bare name is fine when unique; ambiguous names
must be disambiguated as `type:name`.

## Commands

```bash
npm install                       # workspace deps (root + packages/*)
npm run build:registry            # regenerate registry.json from skills/ commands/ agents/  (run after ANY content change)
npm run build:cli                 # build packages/skillpool/dist via tsup

# test the CLI against the local registry without pushing:
node packages/skillpool/dist/index.js --registry . list                 # all types (interactive picker installs selection)
node packages/skillpool/dist/index.js --registry . list --type agent
node packages/skillpool/dist/index.js --registry . add command:commit    # type:name disambiguates
node packages/skillpool/dist/index.js --registry . info <type:name>
```

CLI dev (inside `packages/skillpool/`): `npm run build` (tsup), `npm run dev` (tsup --watch).

There is no test runner and no linter configured.

## Critical workflow rule

**Never hand-edit `registry.json`.** It is generated by `scripts/build-registry.mjs`,
which scans `skills/*/skill.json`, the `commands/*.md`, and the `agents/*.md` files,
recursively lists each skill folder's files, and writes the index. After adding or
editing any resource, run `npm run build:registry` and commit the result. A resource's
declared `name` MUST equal its folder/file name (the build script warns and uses the
folder/file name otherwise).

## Adding a resource

- **skill** тАФ `skills/<name>/` needs a `skill.json` (`{ name, version, description,
  author, tags }`) plus prompt files (`SKILL.md` is the agent system prompt; commonly
  also `README.md`, `examples.md`, `rules.md`). All files in the folder get bundled.
  The `add-registry-skill` skill in `.claude/skills/` automates this end-to-end.
- **command** тАФ `commands/<name>.md`, a Claude Code slash command: YAML frontmatter
  (`name`, `description`, optionally `version`, `author`, `tags`, `argument-hint`,
  `allowed-tools`, `model`) followed by the prompt body.
- **agent** тАФ `agents/<name>.md`, a Claude Code subagent: YAML frontmatter (`name`,
  `description`, optionally `version`, `author`, `tags`, `tools`, `model`) followed by
  the system prompt body.

For commands/agents the build script reads metadata from the frontmatter via a tiny
dependency-free parser; `version` defaults to `0.0.0`, `author` to `ai-skills-registry`.
Then regenerate the index.

## CLI architecture (`packages/skillpool/src/`)

- `config.ts` тАФ registry source resolution. Order: `--registry` flag тЖТ `SKILLPOOL_REGISTRY`
  env тЖТ default GitHub raw base (`jsznpm/ai-skills-registry`). A "source" is a base that
  contains `registry.json` + `skills/` + `commands/` + `agents/`; it can be an http(s)
  URL **or a local path**. `sourceUrl()`/`isRemote()` branch on which. `targetDir(type)`
  maps each `ResourceType` to its install dir (`.claude/skills` | `.claude/commands` |
  `.claude/agents`); the manifest lives at `.claude/skills/manifest.json`.
- `registry.ts` тАФ `readSource()` is the single fetch-or-readFile abstraction over
  remote/local sources; everything else goes through it. `ResourceEntry` carries a
  `type`; `fetchRegistry()` normalizes each array's `type`. `allResources()` flattens
  all three arrays; `findResource(reg, name, type?)` looks up by name and **throws on
  cross-type ambiguity**. `parseSpec()` splits `type:name@version` (type optional).
  `fetchResourceFile()` fetches one file of any resource (`fetchSkillFile` is an alias).
- `installer.ts` тАФ `installResource()` downloads each file to its `targetDir`: skills go
  to `.claude/skills/<name>/`, commands/agents to a single `.md` in their dir. Installs
  are tracked in the manifest keyed by `type:name` (`{ installed: { "skill:docx": {
  type, version, files, installedAt } } }`). `readManifest()` **migrates** legacy
  bare-name (skill-only) manifests to the namespaced shape. `removeResource(key)` deletes
  the folder (skill) or file(s) (command/agent) and drops the manifest entry.
- `picker.ts` тАФ interactive multi-select list (`pickResources`) shared by `list` and the
  no-arg `add`. Reuses `colors.ts`; keys: тЖСтЖУ move, space toggle, `a` all, enter install,
  q/esc cancel. Shows a type tag per row.
- `index.ts` тАФ commander setup. The global `-r, --registry` option is applied in a
  `preAction` hook calling `setRegistry()` before any command runs. Also prints the
  banner once per run via `showBannerOnce()` and registers `BANNER` as help text.
- `colors.ts` тАФ zero-dep ANSI helper (`color.cyan`, `.dim`, `.selected`, тАж). Auto-disables
  when stdout is not a TTY or `NO_COLOR` is set.
- `banner.ts` тАФ ASCII `BANNER` string + `showBannerOnce()`, which prints to stderr on the
  first run only (gated by a `~/.skillpool/seen` marker file) so piped output stays clean.
- `commands/` тАФ one file per command (`list`, `search`, `add`, `info`, `remove`, `update`).
  `list` (TTY) and no-arg `add` open the picker and install the selection; `add`/`info`/
  `remove`/`update` accept `type:name` and resolve via `findResource`/`parseSpec`.

When adding a CLI command: create `commands/<name>.ts`, wire it in `index.ts`, and route
all registry/file access through `registry.ts` (and installs through `installer.ts`) so
local-vs-remote and all three resource types keep working.
