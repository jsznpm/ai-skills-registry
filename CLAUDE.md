# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A monorepo with two halves that share one data file:

1. **The registry** (`skills/`, `registry.json`, `scripts/build-registry.mjs`) — the
   content. Each skill is a folder of prompt files plus a `skill.json` manifest.
2. **The `skillpool` CLI** (`packages/skillpool/`) — published to npm. A package
   manager that installs skills into a consuming project's `.claude/skills/`.

The two are decoupled by **`registry.json`**: an auto-generated flat index that lists
every skill and the exact files it contains. The CLI fetches this index from a GitHub
raw URL at runtime and downloads each listed file by raw URL — it never uses the GitHub
API and never hardcodes the skill list. **Consequence: adding skills never requires a
CLI release.** Add a folder, regenerate the index, push.

## Commands

```bash
npm install                       # workspace deps (root + packages/*)
npm run build:registry            # regenerate registry.json from skills/  (run after ANY skill change)
npm run build:cli                 # build packages/skillpool/dist via tsup

# test the CLI against the local registry without pushing:
node packages/skillpool/dist/index.js --registry . list
node packages/skillpool/dist/index.js --registry . info <skill>
```

CLI dev (inside `packages/skillpool/`): `npm run build` (tsup), `npm run dev` (tsup --watch).

There is no test runner and no linter configured.

## Critical workflow rule

**Never hand-edit `registry.json`.** It is generated by `scripts/build-registry.mjs`,
which scans `skills/*/skill.json`, recursively lists each folder's files, and writes the
index. After adding or editing any skill, run `npm run build:registry` and commit the
result. A skill's `skill.json` `name` MUST equal its folder name (the build script warns
and uses the folder name otherwise).

## Adding a skill

`skills/<name>/` needs a `skill.json` (`{ name, version, description, author, tags }`)
plus prompt files (`SKILL.md` is the agent system prompt; commonly also `README.md`,
`examples.md`, `rules.md`). All files in the folder get bundled. Then regenerate the
index. The `add-registry-skill` skill in `.claude/skills/` automates this end-to-end.

## CLI architecture (`packages/skillpool/src/`)

- `config.ts` — registry source resolution. Order: `--registry` flag → `SKILLPOOL_REGISTRY`
  env → default GitHub raw base (`jsznpm/ai-skills-registry`). A "source" is a base that
  contains `registry.json` + `skills/`; it can be an http(s) URL **or a local path**.
  `sourceUrl()`/`isRemote()` branch on which. Install target is the const `.claude/skills`.
- `registry.ts` — `readSource()` is the single fetch-or-readFile abstraction over
  remote/local sources; everything else (`fetchRegistry`, `fetchSkillFile`) goes through
  it. `parseSpec()` splits `name@version`.
- `installer.ts` — downloads each file of a skill into `.claude/skills/<name>/` and
  tracks installs in `.claude/skills/manifest.json` (`{ installed: { <name>: { version,
  files, installedAt } } }`). `update`/`remove` read this manifest.
- `index.ts` — commander setup. The global `-r, --registry` option is applied in a
  `preAction` hook calling `setRegistry()` before any command runs.
- `commands/` — one file per command (`list`, `search`, `add`, `info`, `remove`, `update`).

When adding a CLI command: create `commands/<name>.ts`, wire it in `index.ts`, and route
all registry/file access through `registry.ts` so local-vs-remote keeps working.
